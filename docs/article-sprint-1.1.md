## 1. 痛点引入

做机器人数据采集的人，十个里有八个踩过同一个坑：数据落盘了，跑训练崩了，查了半天发现是帧率不一致、缺了几帧、或者任务标签对不上。

这不是小众问题。哪怕是 HuggingFace LeRobot 官方维护的数据集，同一套采集流程跑出来的数据，质量也参差不齐——pusht 干净得像教科书，svla_so100_pickplace 的元数据结构就明显粗犷一些。更别说每家公司自己的采集流程，摄像头时钟不同步、ROS topic 丢帧、metadata 全靠手填，进了训练才知道有问题。

根本原因是：**质量检查这个环节，从来都是"人工肉眼看一看"或者"跑个训练试试看就知道行不行"——没有人在数据入库前设一道卡。**

这一次，我在 Sprint 1.1 里用 Claude Code CLI，花了大约 4 小时，从零到把一套完整的数据质量门禁系统跑通、可演示、可交付。

## 3. 门禁设计

### 3.1 5 项检查指标的选择逻辑

不是所有数据问题都同等重要。我从"直接导致训练失败"和"污染模型但不立即崩溃"两个维度，把问题分成四类：

| 类别 | 后果 | 对应检查项 |
|---|---|---|
| 致命 | 训练直接崩或结果 NaN | 缺帧率、帧率一致性 |
| 严重 | 模型学到错误行为，需重训 | 任务标签缺失 |
| 中等 | 多机采集数据偏移，泛化差 | 多视角时间戳偏移 |
| 轻微 | 人工可发现，需额外后处理 | 异常帧（黑屏/模糊） |

基于这个分类，我选了 5 项指标：

- **帧率一致性**：FPS 标准差 / 均值，反映采集时钟稳定性
- **缺帧率**：连续帧序列是否有丢失，直接影响时序连续性
- **多视角时间戳偏移**：多摄像头数据的时间对齐精度
- **任务标签缺失率**：language_instruction 是否完整，直接决定 VLA 训练可用性
- **异常帧比例**：黑屏、过度模糊等低质量帧的占比

### 3.2 strict / loose 双配置

不同场景用不同阈值：

| 指标 | strict | loose |
|---|---|---|
| 帧率标准差 / 均值 | < 5% | < 10% |
| 缺帧率 | < 1% | < 3% |
| 多视角偏移 | < 50 ms | < 100 ms |
| 标签缺失率 | = 0% | < 2% |
| 异常帧比例 | < 0.5% | < 2% |

strict 是交付给客户的正式门禁；loose 是研发阶段快速自检用的。代码只有一处阈值配置，切换只需要改 `--profile strict` 或 `--profile loose`。

### 3.3 数据四级标准

这是我自己定义的规范，写进了项目的 `CLAUDE.md`：

```
raw → validated → curated → training
```

- **raw**：落盘即 raw，不做假设
- **validated**：通过全部 5 项检查，贴上合格证
- **curated**：人工抽检，确认任务描述准确、视角覆盖完整
- **training**：curated 基础上，写入 dataset card（含版本号、缺陷记录）

**铁律：未经 validated 的数据禁止直接用于训练或对外演示。** 这条规则写死在 CLAUDE.md 里，之后所有 Sprint 必须遵守。

### 4.2 发现的数据质量问题

跑门禁的过程中，发现了几个在真实数据上才有的"惊喜"：

**pusht** 虽然干净，但 `meta/info.json` 里没有显式的 FPS 字段，门禁回退到默认 30fps 假设——如果实际采集是 15fps 的设备，这个假设就会导致误判。这是个边界条件，我在 `_human_size` 函数里用内联替代了外部 humanize 依赖，减少了潜在问题。

**so100_pickplace** 的视频文件分布在两个目录（`observation.images.top` 和 `observation.images.wrist`），但 parquet 里实际只记录了单一视角的数据字段——多视角目录存在但数据为空。这意味着：如果将来采集端真正启用多视角，这套门禁的多视角同步检查立即可以接管，不需要改代码。

这正是门禁的价值：**数据质量问题早发现，修复成本低；进了训练才发现，代价是整轮重训。**

## 5. Claude Code 体验

### 5.1 工作模式

用 Claude Code CLI 做工程开发，我摸索出一套"人类下达 → AI 执行 → 人类审 diff → 自检审计"的工作闭环：

1. **人类下达**：用中文描述需求，附上 CLAUDE.md 里的规范
2. **AI 执行**：CC CLI 生成代码，逐条展示 diff，等确认
3. **人类审 diff**：关键文件（CLAUDE.md、阈值配置、退出码语义）必须逐字核对
4. **自检审计**：用专门的自检 Prompt，逐条对 CLAUDE.md 举证，超200行、路径硬编码、依赖未登记，一项都不放过

### 5.2 自检审计：抓"假装合规"

CC CLI 在高强度生成时，偶尔会出现"自认为合规但实际违规"的情况。典型案例：

- 生成代码时用了 Python 关键字 `pass` 做 dict key 名称，导致运行时 `SyntaxError`
- 在 f-string 里写了带反斜杠的表达式，Python 3.9 报错
- 用 `is True` 做 numpy bool 的 identity check，单测失败

这些问题都是自检审计抓出来的。CLAUDE.md 自检 Prompt 模板：

```
请重新完整读一遍项目根目录的 CLAUDE.md，然后逐项审计你刚才的产出，
带证据举证。任何"我应该是做了"的回答都不接受。
```

这套自检模板本身就是可复用资产，我已经把它固化进了 CLAUDE.md，供后续 Sprint 直接调用。

### 5.3 边界情况处理

门禁代码对缺失数据的处理原则：**保守通过**。当 meta 目录结构不完整时，检查模块不会报错，而是返回 `passed=True` 并在 details 里注明原因。这样设计是因为：

- pusht 没有 FPS 字段 → 默认 30fps，保守通过
- so100 无多视角数据 → multiview_sync 跳过，保守通过
- 空 episodes 目录 → 返回 `note: no episode data found`

这不是完美方案，但它是**诚实的**：不做假设、不强行报错、把判断留给人类。这符合我定的那条输出风格原则：**给数据，不给情绪；给建议，附上"为什么"。**
